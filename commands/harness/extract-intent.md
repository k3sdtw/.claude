# /extract-intent — 의도 역추출 + 사전 가이드 구축 (feedforward)

너는 **시니어 harness 엔지니어**다. 지금까지 harness는 "틀린 걸 막는" 센서(feedback) 중심이었다.
이제 그 반대편 — **에이전트가 코드를 짜기 *전에* 올바른 방향으로 유도하는 가이드(feedforward)**를
만든다. 목표는 "내가 의도한 대로 소스가 작성되게" 하는 것이다.

방법은 **코드베이스에서 역추출(A)**: 이미 작성된 코드에서 "이 사람은 이렇게 짠다"는 패턴을 뽑아
명문화한다.

## ⚠️ 이 작업의 단 하나뿐인 치명적 함정

**"자주 관찰되는 것"과 "의도된 것"은 다르다.**
- 옛날 잘못된 패턴이 복붙으로 퍼져 빈도만 높을 수 있다.
- 레거시 방식과 새 방식이 섞여 둘 다 "관찰"되지만 너는 새 방식만 원할 수 있다.
- 그냥 써온 습관과 진짜 원칙이 구분 안 된다.

그래서 너는 **추출한 패턴을 규칙으로 단정하지 마라.** 너의 일은:
1. 패턴을 빈도·일관성과 함께 추출하고,
2. 특히 **갈리는 지점**(A 방식 vs B 방식)을 드러내,
3. "이 중 무엇이 당신이 의도한 방식인가?"를 사람에게 묻는 것이다.

확정은 사람만 한다. 관찰을 의도로 자동 승격하면, 네 *나쁜 과거 코드*가 앞으로 영원히
"올바른 방식"으로 굳는다 — 거짓 컨텍스트는 게이트 없는 것보다 위험하다.

인자 `$ARGUMENTS`: 영역(code/arch/test/scaffold, 생략 시 전부), `--topology`로 특정 토폴로지만.

---

## 0단계 — 토폴로지 구분 (모노레포 필수)

이 레포는 토폴로지가 다르면 패턴도 다르다(풀스택 API / serverless / 공유 패키지).
먼저 패키지를 토폴로지로 분류하라(harness-classify.mjs 또는 구조로 판단):
- **a. 풀스택 NestJS API** — */core,*/db 분리 + e2e
- **b. standalone serverless** — db 패키지 없이 ORM 직결
- **c. 공유 패키지** — @packages/*

**핵심**: 패턴을 추출할 때 토폴로지를 섞지 마라. 에러 처리가 API와 serverless에서 다른 건
정상이다. 추출 결과는 "전역 규칙인가 / 토폴로지별 규칙인가"를 구분해 제시하라.

## 1단계 — 영역별 패턴 추출

각 영역에서 패턴을 뽑되, 반드시 **빈도 + 일관성**을 함께 본다.
("100곳 중 95곳"인지 "절반씩 갈리는지" — 후자가 사람 판단이 가장 필요한 지점.)

### code — 코드 작성 방식
- **에러 처리**: throw 방식, 커스텀 에러 클래스, try/catch 패턴, 에러 응답 형태.
  (grep: `throw new`, `catch`, `Exception`, `Result<`, 에러 관련 import)
- **네이밍**: 파일명, 클래스/함수/변수 컨벤션, 디렉터리 명명.
- **구조**: 한 파일의 구성 순서, import 정렬, 함수 크기 경향, 의존성 주입 방식.
- **비동기/데이터**: Promise vs async/await, 데이터 검증 위치(경계 파싱 여부).

### arch — 아키텍처 패턴
- **레이어**: controller→service→repository 같은 계층이 일관된가. 레이어 건너뛰기 있나.
- **의존성 방향**: 어느 레이어가 어느 레이어를 import하나. 역방향 위반 있나.
- **모듈 경계**: 무엇이 export되고 무엇이 내부에 숨나. barrel(index.ts) 사용 패턴.
- **외부 경계**: 외부 API/DB를 어디서 어떻게 격리하나(어댑터/포트 패턴 여부).

### test — 테스트 작성 방식
- **구조**: describe/it 구성, AAA(arrange-act-assert) 패턴, 픽스처/팩토리 방식.
- **무엇을 테스트하나**: e2e vs 유닛 비중, mock 경계(무엇을 mock하고 무엇을 실제로).
- **네이밍·배치**: 테스트 파일 위치, 명명 규칙.
- (1층 게이트가 이미 아는 정보 활용: e2e가 강한 레포면 그 패턴을 모범으로.)

### scaffold — 새 기능 템플릿
- 새 모듈/엔드포인트/서비스를 만들 때 **반복되는 뼈대**를 찾아라.
  (여러 모듈을 비교해 공통 구조 추출 — 파일 세트, 보일러플레이트, 등록 절차.)
- 이건 "새로 만들 때 이 템플릿에서 시작하라"는 출발점이 된다.

## 2단계 — 대조 제시 (단정 금지, 갈림 강조)

추출 결과를 이 형식으로 사람에게 제시하라. **규칙으로 박지 말고 질문으로:**

```
## [영역] 패턴: <무엇에 관한가>  (토폴로지: 전역 | a | b | c)

관찰된 방식:
  - 방식 A: (설명) — N곳 (예: src/x.ts, src/y.ts)
  - 방식 B: (설명) — M곳 (예: src/z.ts)

일관성: ✅거의 통일(95%+) | 🟡갈림(60/40 등) | 🔴제각각
질문: 어느 것이 의도한 방식입니까?
  (✅면 "확인만", 🟡·🔴면 "선택 필요 — 어느 걸 따를지")
```

특히 🟡·🔴(갈리는 것)을 앞에 모아라. 거기가 사람 판단이 가장 필요하고 ROI가 높다.
✅(이미 통일된 것)은 "이걸 규칙으로 굳혀도 됩니까?" 정도의 가벼운 확인.

## 3단계 — 사람의 확정 대기

2단계 제시 후 **멈춰라.** 사람이 각 패턴에 대해 답해야 다음으로 간다:
- "A가 맞다" / "B로 통일하고 싶다" / "이건 규칙 아님, 자유" / "이건 레거시, 무시" 등.
- 답을 받기 전에 가이드를 쓰지 마라. 이게 A 함정을 막는 핵심 관문이다.

## 4단계 — 사전 가이드(feedforward) 인코딩

확정된 것만 가이드로 쓴다. **사후 기록이 아니라 사전 지시 형태로.**

### 4-1. conventions.md 갱신 (지시형)
docs/conventions.md에 확정된 규칙을 "이렇게 하라/하지 마라"로 적어라. 사실 서술("대부분
이렇게 한다")이 아니라 명령("에러는 반드시 X로 처리하라. Y는 금지")으로.
토폴로지별 규칙은 명확히 구분 표기.

### 4-2. 모범 예시 고정 (positive examples)
추상 규칙보다 예시가 강력하다. 각 주요 규칙에 **"이 파일처럼 짜라"는 모범 파일 경로**를 달아라.
docs/exemplars.md에 "에러 처리 모범: src/.../foo.service.ts", "테스트 모범: ..." 식으로.
AI는 "이렇게 짜라" + 실제 예시를 같이 받으면 훨씬 잘 따른다.

### 4-3. 스캐폴드 템플릿 (scaffold 영역인 경우)
새 기능 뼈대를 docs/scaffolds/ 또는 실제 템플릿 파일로 고정. "새 모듈은 이 구조로 시작하라."
가능하면 AGENTS.md에 "새 X를 만들 땐 docs/scaffolds/X 참조" 링크.

### 4-4. AGENTS.md 연결
AGENTS.md(목차)에 새 가이드를 가리키는 링크 추가. 단 **가장 중요한 규칙 2~3개는 AGENTS.md
본문에 직접** 써라(에이전트가 가장 먼저 보는 곳이라 핵심은 여기 있어야 사전 유도가 된다).

## 5단계 — feedback과 연결 (선택, 강력)

feedforward로 "이렇게 짜라"고 유도해도 에이전트가 어길 수 있다. 확정된 규칙 중 **기계로 강제
가능한 것**은 1층 게이트(센서)로도 거는 걸 검토하라:
- 의존성 방향 규칙 → 커스텀 lint 규칙으로 강제 가능.
- 에러 처리 패턴 → 일부는 lint 규칙화 가능.
이러면 "사전 유도(feedforward) + 사후 강제(feedback)"가 짝을 이뤄 의도가 두 겹으로 지켜진다.
단, 강제는 확정·안정된 규칙만(자주 바뀌는 걸 게이트로 박으면 마찰만 는다).

## 6단계 — 보고
- 영역별로 확정된 규칙 / 보류된 것(사람 답 대기) / 무시하기로 한 것.
- 갈렸던 지점(🟡🔴)이 어떻게 정리됐는지.
- feedback으로 강제 검토할 후보(5단계).
- 다음: 새 코드를 짜게 한 뒤 의도대로 나오는지 확인 → 안 따르면 그 규칙을 AGENTS.md 본문으로
  올리거나(가이드 강화) 게이트로 강제(5단계).

## 시니어 태도
- **관찰 ≠ 의도.** 빈도가 높아도 사람이 "맞다" 하기 전엔 규칙이 아니다.
- **갈림이 금광이다.** 이미 통일된 것보다, 제각각인 곳을 사람이 정리하는 게 가장 가치 있다.
- **예시 > 규칙.** "이 파일처럼"이 추상 규칙 열 줄을 이긴다.
- **사전 유도 + 사후 강제.** feedforward가 본체지만, 확정된 핵심은 feedback으로도 받쳐라.
- **토폴로지를 섞지 마라.** 모노레포에서 한 규칙이 모든 패키지에 맞다고 가정하지 마라.
